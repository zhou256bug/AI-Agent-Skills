## 一、海外经销合同审核

### 输入

菜头发送合同文件 + 口头说明客户背景（国家、客户名、合作历史）。

### 委派（大为侧）

法律条款深审委派 **法务方律**（profile `fanglv`），大为整合 🔴/🟡/🟢 意见后向菜头交付。区域最低采购量合理性可并行委派通哥（`tongge`），法律结论以方律为准。

**委派语法**（与大为 `SOUL.md` §委派命令 一致；**无** `profile run` / `profile use`）：

1. **Gateway**：`delegate_task(goal=…, context=…, toolsets=[…])`
2. **CLI**：`hermes -p fanglv chat -q "…"`

示例：
```
delegate_task(
  goal="你是方律。审核合同并输出 🔴🟡🟢 + redline",
  context="profiles/fanglv/SOUL.md；hzr-contract-routing；Playbook 路径；合同文件路径",
  toolsets=["terminal","file","search"]
)
```

**无论用哪种方式，都要在 `context`/`goal` 中明确指定：**
- 合同类型（经销/采购/用工·国别）
- Playbook 完整路径（前缀 `business/huazhirong-business-support/references/`）
- 方律身份：`profiles/fanglv/SOUL.md`（可选，复杂任务建议附上）
- 客户/供方名、国家、背景

**路由**：`business/huazhirong-business-support/references/hzr-contract-routing.md`

**Playbook**：
- 海外经销（我方乙方）→ `.../hzr-negotiation-playbook.md`
- 供应商采购（我方甲方）→ `.../hzr-procurement-playbook.md`
- 用工·中国 → `.../hzr-labor-contract-playbook.md`
- 用工·境外 → `.../hzr-labor-law-jurisdictions.md`

**辅助技能**：`contract-review-anthropic`、`tabular-review-lawvable`  
归档：经销 `newpos/合同/审核记录/`；采购 `newpos/合同/采购/审核记录/`；人事 `newpos/合同/人事/审核记录/`。

### Step 1: 合同全文提取

- `.doc` 文件：`textutil -convert txt <file> -output /tmp/contract.txt`
- `.docx` 文件：`python3 -c "import docx; print(docx.Document('<path>').text)"`
- `.pdf` 文件：用 OCR skill 或 pymupdf 提取

**⚠️ 长PDF（10+页）的三遍阅读策略**：一次性读完超长合同会丢失边界信息。推荐以下分遍策略：

| 遍数 | 读取范围 | 目标 |
|:----|:---------|:-----|
| 第一遍 | Header（合同主体、签署方、日期）+ Scope + Term | 搞清楚"谁、什么产品、多久" |
| 第二遍 | Remuneration + Termination + IP + Governing Law | 搞定"钱、退出、法律底线" |
| 第三遍 | Warranties + Liability + Force Majeure + Notices + Annexes/Schedules | 验收细节和缺失项 |

**⚠️ 附件（Annex/Schedule）检查**：提取完整文本后，**必须检查文件末尾是否有附件**（通常以 Annex / Schedule / Appendix / Anexo 开头）。如果附件被遗漏（比如 PDF 扫描页），先问菜头是否有附件。**不要自行假设附件不重要**——附件常包含产品型号、价格清单、SLA 等核心商业条款。

**⚠️ 双语合同处理**：拉美/欧洲合同常为英/葡/西双语。提取后注意：
- 双语合同篇幅加倍（每段/每句同时出现在两种语言中）
- 以英文版为主进行提取和分析，第一遍用英文完成评估
- 对关键条款（独家范围、终止条件、管辖法）确认葡/西语版与英文一致
- 如果发现语言冲突条款（如"如英葡版本有冲突以葡版为准"），标记为 🔴 高风险

### Step 2: 历史合同查找

菜头经常和同一客户续签。务必先查历史合同。

**查找顺序**：
1. Obsidian vault → 客户名 → `客户维护/客户信息/`
2. `newpos/` 目录下搜索同名客户旧协议
3. 菜头发来的旧文件直接对比

**对比结构**：最低采购量 / 独家区域 / 付款条件 / 保证金 / 管辖法律+仲裁 / 法律登记条款 / 产品范围

**⚠️ 重点标记：新增的本地法条款** — 对于老客户续签，最危险的改动是对方新增当地保护性法律条款（如多米尼加 Law 173、巴西代理商保护法等）。属于 P0 底线问题，必须删除或大幅修改。

### Step 3: 三类条款评估

**🔴 高风险 — 必须修改**：
1. 独家范围与最低采购量的匹配度 — 菜头经验法则：一家公司能拿区域总需求的15%已经是极限
2. 本土保护法条款（多米尼加 Law 173、巴西代理商法等）
   - **关键判断**：对方要求引用当地法，不是想绑死你，而是想让合同在本国有法律执行力
   - **菜头方法论**：接受引用当地法来满足对方诉求，但同时写入明确的终止触发条件（见下方 note）
   - **底线**：不得接受"授权在主管机构登记合同"而无对应终止条件
3. "最优惠价格"条款 — 建议限缩为"区域内最优惠"
4. 知识产权授权范围 — 是否明确授权，是否限于"推广销售之目的"

> **📝 本土保护法的谈判策略（菜头方法）**
> 1. 接受引用当地法 — 满足对方"合同在当地有法律效力"的核心诉求
> 2. 同时写入明确的终止触发条件：
>    - 年度最低采购量不达标 → 有权终止独家权
>    - 连续两季度未完成采购量70% → 有权终止
>    - 逾期付款超30天 → 有权终止
>    - 其他违约经通知未整改 → 有权终止
> 3. 加一句"本地法不影响本合同约定的管辖法和仲裁条款"
> 这样当地法变成保护双方的框架——对方有法律承认的地位，我方有清晰量化的退出路径。

**🟡 中风险 — 值得注意**：竞业限制、质保条款、赔偿上限、终止后库存处理

**🟢 低风险 — 可接受**：T/T 预付、FCA/FOB 术语、中国法律+华南国仲、保密条款

### Step 4: 市场验证（可选）

- 搜索国家名 + "POS terminals market" / "payment market"
- 菜头15%法则：能拿15%是极限
- 输出表格：国家 / 年需求估算 / 15%目标量 / 合同最低量

### 交付格式

结论先行。按 🔴必须改 / 🟡注意 / 🟢可接受 分段。

### 菜头偏好

- **15%法则**：先算15%目标量再对比合同最低量
- **简洁结论先行**：直接说"最大问题是..."再展开
- **不重复说理**：同一判断说一次
- **新旧合同直接对比**：不需先问要不要对比
- **历史合同是基线**：对于续签客户，先找到老合同（如2018年版本），把它当作"干净基线"。本地法条款（如Law 173）如果是新版才出现的，要特别标注"这是新版新增的，不是沿用旧约"——这直接影响谈判策略

## 二、客户档案管理

### 触发条件

菜头要求为某个客户/公司创建档案，或补充/更新现有客户信息。

### 工作流程

**Step 1：查询公开信息** — 官网、新闻、行业报告、合作动态。

**Step 1a：多源发现管线（从CSV/名单起步）**
当菜头仅提供一个客户名（如"Capitec"）开始建档案时，按以下管线增量搜资：

1. **查内部名录** → `客户列表.csv`（newpos/销售/）确认客户存在、地区、销售负责人
2. **查Obsidian** → 客户信息总表、已有客户档案
3. **名/别称确认** → 如有拼写误差（如"Capitec"是正确名），先确认再继续
4. **Web搜索** → 公司概况、市场地位、规模
5. **LinkedIn搜索** → 代理/中间商负责人（如Tian van Zyl / SCOTCH. Software）
6. **邮件搜索** → 历史订单记录、往来邮件
7. **名片处理** → 见 `business/emma-customer-maintenance` §一 + `productivity/business-card-crop-and-ocr`：**大为**视觉确认并 **OCR**、原图存 `附件/普通附件/` → **委派艾玛**写 `客户维护/客户名片/{大洲}/{国家}/{国家}-{公司}-{姓名}.md` 并 patch 档案联系人（**艾玛禁止 OCR**）→ 大为 `audit_card_vault` 后回复菜头
8. **会议纪要** → 飞书妙记 → 转录全文提取 → 委派艾玛整理 → 存 `客户维护/Meetings/` → 双链回客户档案

**增量原则**：先查已有的，不够再搜外网。不自创信息。

**Step 1a：先调查现有笔记格式（2026-06-02 菜头纠正后固化）**

在创建任何新客户笔记前，必须先查看同目录下的已有笔记：
1. 用 `ls` 命令列出 `客户维护/客户信息/` 目录，了解命名惯例
2. 用 `read_file` 读取 1-2 个已有笔记（如 `巴西-Pagbank.md`），确认 YAML 属性、正文结构、表格格式
3. 不自创格式——已有笔记就是模板，参照最完整的那个
4. 如果菜头说"参照其他笔记"或指出格式不对，说明跳过了这个步骤

**路径一致性铁律（2026-06-09 菜头纠正）**：客户信息目录（`客户信息/`）与客户名片目录（`客户名片/`）的**大洲/国家层级必须完全一致**。建新国家第一条时，先查名片目录的结构：`ls 客户维护/客户名片/`。名片用 `亚太/韩国/` → 客户信息也必须用 `亚太/韩国/`，不能直接用 `韩国/`。

**国家前缀**：参照 `references/latam-country-prefix-lookup.md`，不要自创国家前缀。

**Step 2：参照现有客户档案的格式**

客户文书类任务委派艾玛时，**必须**指定技能 `emma-customer-maintenance`，不得让艾玛自创 YAML/正文结构。

不得自创属性名称。必须参照已有档案（如`巴西-Pagbank.md`）的完整格式：

frontmatter 必须使用的属性：
```yaml
---
区域:
  - 国家名
  - 大洲/区域名
日期: YYYY-MM-DD
客户性质:
  - 公司类型
封面: ""  # Logo图片双链，用"[[图片文件名|公司名Logo]]"格式，暂无则留空
tags:
  - 客户
  - 国家名
---
```

正文结构（参照 Pagbank 笔记）：
```
## 客户简介
**公司全称（注释说明）** 一句话概括公司性质、核心业务、市场地位。

![[封面图片.png]]  # Logo图片，可选

#### 一、合作情况
[[双链到具体Meeting笔记或交流笔记#段落]]
- 时间线记录合作进展

#### 二、核心信息速览

| 项目 | 详情 |
|:--|:--|
| **公司全称** | ... |
| **成立 / 起步** | ... |
| **总部** | ... |
| **监管资质** | ... |
| **核心业务** | ... |
| **关键数据** | 用户数、商户数、交易额等 |
| **现有终端** | 当前在用品牌+型号+价格 |
| **官网** | ... |
```

**数据来源原则**：正文中的公司简介需基于公开搜索到的资料（官网、维基、Crunchbase、新闻），不能只依赖Risto报告中的简介段落。合作情况来自出差报告或会议纪要。

**Step 3：联系人双链规则** — 用`[[名片文件名|📇名片]]`双链指向名片笔记文件，不重复写联系方式。

**Step 4：写入流程** — **委派艾玛**（`hermes -p emma chat -q "…" -Q -s emma-customer-maintenance`）。大为只做公开信息查询与结构化提取，**禁止**自己 `write_file` 客户档案/名片。大为 audit 后回复菜头。

**Step 5：创建后验证** — 确认文件存在、YAML格式正确、双链可解析。

### Pitfalls（2026-06-02 菜头纠正后更新）

- ⚠️ **不自创 frontmatter 属性**（只用`日期`/`区域`/`客户性质`/`封面`/`tags`）
- ⚠️ **必须包含 YAML 属性**（不要只写纯文本简介）
- ⚠️ **必须有合作情况区块**（`### 一、合作情况`），不能只写公司简介
- ⚠️ **必须有核心信息速览表格**（`### 二、核心信息速览`），不能只有文字段落
- ⚠️ **封面字段留空不删**，后续补Logo
- ⚠️ 联系人只用双链，不重复写联系方式
- ⚠️ 不要删文件（菜头铁律）
- ⚠️ 写入后验证文件确实存在

## 三、品牌命名评估

### 触发条件

菜头问一个品牌名/产品名"是否合适""会不会被联想到XX""评估一下"。如 NewPostH、产品线名称等。

### 评估框架

从四个维度分析：

1. **行业锚点** — 这个名字让人第一时间联想到POS/支付行业吗？PAX→支付代名词，Verifone→Verify+Phone一听就是支付验证，Newland→"新大陆"有开拓感但有科技辨识度。如果和行业无关，需要额外教育成本。

2. **已有品牌混淆风险** — 搜索是否有同名品牌（加拿大Newpost是新闻平台、华为H系列等）。名字是否处于被大品牌"压着"的联想空间。

3. **故事性** — 是否有好讲的故事？"新大陆""百富"都有故事可讲。纯组合词（NewPostH）故事性弱，需要额外营销投入。

4. **简洁与记忆度** — 多长？几个音节？是否容易拼写和发音？能否成为动词/代名词？

### 交付格式

结论先行。逐维度分析，最后给一个明确判断（可/不可/需要配合营销教育）。

### 参考竞品

| 品牌 | 风格 | 优劣 |
|------|------|------|
| PAX（百富） | 4字母无意义 | 短、易记、已成支付代名词 |
| NEWLAND（新大陆） | 寓意型 | "新大陆"寓意开拓，好故事 |
| Verifone | 功能型 | Verify+Phone，一听就懂 |
| NewPostH（评估示例） | 混合型 | 8字母、支付感弱、H含义模糊 |

## 四、公司/个人背景调研

### 触发条件

菜头说"你去网上查一下XX，以便你更了解我"或类似意思。

### 工作流程

**Step 1：明确调研范围** — 公司层面 / 行业层面 / 个人层面。不需要问菜头细节。

**Step 2：多渠道搜索** — 至少3-4轮递进搜索：
- 第一轮：宽泛搜索（公司名+IPO/融资）
- 第二轮：定向搜索核心文章
- 第三轮：补充搜索（财务数据、股权结构、行业排名）
- 优先权威来源（官方文件 > 正式媒体报道 > 知乎/自媒体）
- ⚡ **搜索引擎不可用时**：做3次 web_search 尝试都失败后不再重试，切换到 IR 官网直接浏览（先确认目标人员在法定高管还是运营管理层），然后告知用户当前搜索不可用。参见 `references/corporate-executive-background-research.md` §第五步 的备选方案表。

**Step 3：结构化整理** — 存入 Obsidian `AI与运维/` 目录，每条标注来源，区分"事实"和"推测"。

**Step 4：存储记忆** — 关键事实摘要写入 memory（空间不足只存最核心的几行）。

**Step 5：汇报给菜头** — 结论先行，简洁语风。

### 专项：中国企业工商注册状态查询

当需要验证某个公司在中国大陆（如深圳）的注册状态时，使用独立的查询流程，详见 `references/company-registration-verification.md`。这与一般背景调研不同：重点是多渠道交叉验证，结论是 **已注册（含信用代码/法定代表人/状态）** / **未查到注册记录（含排除的近似公司）** 两级，而非写调研笔记。对未找到记录的情况，需说明已排除的近似公司名称以消除混淆。

**关键区别**：
- 背景调研 → 写调研笔记存入 Obsidian
- 工商注册查询 → 简洁口头汇报，不写笔记

### 专项：上市公司高管背景调研

当需要查找特定公司高管（如Cielo的CFO、法务总监等）的简历时，使用独立的搜索和分析流程，详见 `references/corporate-executive-background-research.md`。

**适用场景**：需要确认某高管的职务、教育背景、过往履历时使用。

**与一般背景调研的关键区别**：
- 一般背景调研 → 写调研笔记存入 Obsidian
- 高管背景调研 → 直接用中文概括汇报，不写长笔记
- 上市公司法定高管 → 优先从IR官网提取（有法定披露义务）
- 运营管理层（Superintendent/Director级别）→ IR官网无披露，需LinkedIn

### Pitfalls

- ⚠️ 别只搜一轮（至少要三轮递进）
- ⚠️ 别跳过招股书（A股申报过的公司，招股书最权威）
- ⚠️ 别混淆"主动调研"和"应要求查询"（菜头说"查一下以便了解"是主动学习需求）

用工做重点 sheet 的项目列表为骨架，逐项从转录中提取该项目对应的会议讨论内容。对于首次会议，Excel 基准 = 会议前的状态，转录 = 会议上的讨论更新。

格式：每条项目一行概括，包含「当前状态 + 风险/问题 + 下一步」。

**Step 4: 委派艾玛生成纪要**

用 `hermes -p emma chat -q "…" -Q -s emma-customer-maintenance --accept-hooks` 委派（**禁止** delegate_task）。

指定以下信息：
- 会议名称、时间、时长
- 飞书妙记转录全文（可抄录核心段落）
- Excel 工作重点 sheet 的全部项目及当前状态
- 首次创建 vs 第N次（首次无上文对照，后续需列出与上次对照的进展）

**Step 5: 纪要格式（按类型）**

**管理 / 销售 / 采购例会** — 叙述分节（`#### 一、…`），参照 `公司管理例会2026-06-03.md`、`公司销售例会2026-04-30.md`。

**研发例会** — 六要素结构（菜头标准格式）：

```
YAML：日期 + tags（内部会议、公司研发例会）；可选 来源

一、项目总览（10个项目，按优先级排列，标注自研/外包）
二、各项目详情（逐条：优先级、研发类型、推进决策、风险点）
三、补充订单/其他事项
四、平台与战略决策
五、重要决议表
六、待办事项表（责任人+截止时间）
```

六要素定义：
- **项目优先级**：杨总或菜头敲定的排名，标注P1/P2
- **研发类型决策**：内部自研 or 外包
- **推进决策**：关键决议（如锁定平台、暂缓某市场、外观外包等）
- **风险点**：⚠️ 标注，如认证周期、资源料兼容性、人员紧缺
- **其他事项**：订单数据、管理要求、外包分配等
- **待办事项**：编号表格，含负责人和截止时间

**Step 6: 审核后通知菜头**

- 确认文件确实写入指定路径
- 用 `ls -la` 验证文件存在
- 简要汇报：纪要文件名、项目数量、TODO数量
